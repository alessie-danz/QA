3 Lógica de Evaluación

Reglas de estado

EstadoCuándo se asignaEjemplo real
PASSLa condición principal se cumple dentro del timeoutURL contiene /transferencias en <15s
FAILLa condición no se cumple — el sistema respondió incorrectamente o elemento esperado no aparecióTimeout: formulario no cargó en 15s
ERRORExcepción Python durante ejecución — test no completópage.fill() lanzó PlaywrightError
BLOCKEDPrerequisito de datos no disponible en QA (sin tarjeta, sin beneficiario, funcionalidad desactivada)Sin tarjetas registradas — TC-005 Tarjetas
EXPECTEDSistema tomó flujo alternativo válido y documentado (no bug)Municipio sin paso 2 — va directo a OTP

Mecanismo interno BLOCKED/EXPECTED en consola vs HTML

ℹ️
EnhancedLogger.record() mapea internamente BLOCKED→SKIP y EXPECTED→SKIP cuando llama a self.log() (línea 183 de qa_helpers.py). Por eso la consola muestra ⏭️ [SKIP] para ambos. El HTML los muestra correctamente como BLOCKED/EXPECTED porque el runner usa elog.get_records() que preserva el estado original.

4 Problemas Detectados
Alta Problema 1 — TC-005 Cuentas: Falso PASS (click en elemento incorrecto)

El selector CSS [class*="cuenta"],[class*="card"],[class*="product"] es demasiado amplio. Hizo click en "Agregar Beneficiario Interno" y registró PASS. Ese elemento no es una cuenta bancaria. El TC nunca verificó QUÉ fue clickeado ni si la navegación resultante era la esperada.

Consecuencia encadenada: TC-006, TC-007 y TC-008 se ejecutaron en el estado incorrecto (pantalla de agregar beneficiario). Sus resultados BLOCKED podrían ser consecuencia de este error de navegación, no de funcionalidades ausentes.

❌ Antes
Click en primer elemento con clase "card", "cuenta" o "product" → si clicked → PASS sin verificar destino
✅ Después
Después del click: verificar que URL cambió a /cuentas/{id} O que main_text contiene 'Movimientos' o 'Saldo disponible'. Si no → BLOCKED
Media Problema 2 — TC-010/011/012 Tarjetas: EXPECTED semánticamente incorrecto

Están marcados EXPECTED con "Flujo de pago no ejecutado — TC-009 fue BLOCKED". EXPECTED significa que el sistema tomó un flujo alternativo válido. En realidad, estos casos no corrieron porque el prerequisito (TC-009) estaba BLOCKED — el flujo nunca comenzó.

❌ Antes
EXPECTED — "Flujo de pago no ejecutado — TC-009 fue BLOCKED"
✅ Después
BLOCKED — "Prerequisito no ejecutado — TC-009 fue BLOCKED. Dependiente de selección de tarjeta (TC-005)"
Media Problema 3 — TC-004 Cuentas: OR con keyword '$' trivialmente verdadero

La condición usa any(k in body.innerText ...) donde '$' es una de las keywords. El símbolo $ aparece en la barra de navegación, el header de saldo global u otros elementos fuera del módulo de cuentas, haciendo que la condición sea prácticamente siempre verdadera independientemente de si el módulo cargó.

Adicionalmente, el número de cuenta '9139' está hardcodeado — si cambia, el TC falla.

❌ Antes
any(['$','AHORROS','Cta.','9139',...] in body.innerText)
✅ Después
'$' AND 'AHORROS' AND 'Cta.' en main_content() únicamente (no nav/footer)
Baja Problema 4 — TC-003 Transferencias: texto "monto" como proxy débil de carga

wait_text('monto', 15) verifica que la palabra "monto" aparezca en body.innerText. Esta palabra podría aparecer en el nav lateral, en un mensaje de error, o en el footer — no garantiza que el formulario esté visible y listo.

❌ Antes
wait_text('monto', 15) en body.innerText completo
✅ Después
Verificar existencia de input[name="amount-transfer"] en DOM — ese selector es específico del formulario de transferencias
Baja Problema 5 — TC-008 Transferencias: ID de screenshot incorrecto en rama FAIL

En la rama FAIL del TC-008, el código usa await self.ev.error(self.page, "TC-007") en lugar de "TC-008". Si el caso falla, el screenshot se genera con nombre transferencias_TC-007_error_...jpg en lugar de TC-008_error. El archivo de evidencia queda mal nombrado.

❌ Antes
shot_err = await self.ev.error(self.page, "TC-007")
✅ Después
shot_err = await self.ev.error(self.page, "TC-008")
Alta Problema 6 — Cobertura negativa: 0 casos negativos fuera del módulo Login

Solo existen 3 casos negativos (TC-LN-01/02/03), todos en Login. Los módulos Transferencias, Cuentas, Pago de Servicios y Tarjetas no tienen ningún caso negativo. Los bugs más frecuentes en producción (montos inválidos, campos vacíos en formularios de pago, CCI incorrectos) no se detectarían con la suite actual.

Casos negativos que deberían existir: Monto cero en transferencia · Monto mayor al saldo · Motivo vacío · Cédula de servicio inválida · CCI bancario con dígitos incorrectos

Media Problema 7 — TC-005 Cuentas: BLOCKED retorna True — dependientes corren en estado incorrecto

Cuando TC-005 no encuentra una cuenta para seleccionar, registra BLOCKED pero retorna True. El orquestador run() continúa ejecutando TC-006, TC-007, TC-008 en un estado de página que no es el esperado por esos TCs (pantalla de cuentas sin cuenta seleccionada, o peor, pantalla de agregar beneficiario).

Baja Problema 8 — TC-004/TC-005 Servicios: PASS sin funcionalidad real

TC-004 pasa con Botones: [] porque encontró main_text > 60 chars. Un módulo vacío sin ningún elemento interactivo debería distinguirse entre "módulo cargó funcionalmente" (PASS) y "módulo cargó pero vacío" (BLOCKED). El umbral de 60 caracteres de texto es demasiado permisivo.

Errores comunes a evitar

❌
No definir el estado inicial: "navega a transferencias" es ambiguo si no se sabe si el test debe hacer click en el nav o si ya está ahí como prerequisito de un TC anterior.
❌
Expected ambiguo: "funciona correctamente" no es un expected. Debe ser una condición verificable: URL específica, texto específico, estado del DOM observable.
❌
Prerequisito implícito: si el TC depende de que haya una cuenta seleccionada (TC-007 Transferencias), decirlo explícitamente como prerequisito.
❌
Validar lo que el framework garantiza: no vale probar que page.fill() funcionó — eso es responsabilidad de Playwright. Validar el resultado de negocio: que el valor persiste en el DOM del componente React.
❌
OR sin criterio claro entre condiciones no relacionadas: combinar "existe botón" OR "texto contiene X" OR "inputs > 0" produce PASS cuando solo uno de esos estados es el correcto.
